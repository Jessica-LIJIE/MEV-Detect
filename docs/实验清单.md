## E1：算法替换 — PSO vs GA

### 目标

证明向量化 PSO 相对 GA 消除了串行瓶颈，粒子数增加时耗时增长更平缓。

### 输入

- `data/mock_mempool.json`（固定种子 `seed=42`）
- 粒子数：100 / 500 / 1000 / 5000
- 设备：本地 CPU 或单卡 CUDA（报告中注明）

### 命令

```bash
python tests/test_pso_vs_ga.py
```

### 产出


| 文件                           | 内容                 |
| ---------------------------- | ------------------ |
| `data/figures/pso_vs_ga.png` | 耗时柱状图 + 加速比        |
| 终端日志                         | 各粒子数下 PSO/GA 毫秒耗时表 |


### 验收标准

- PSO 核心循环无粒子级 `for`
- 至少 4 组粒子数对比数据
- PSO 耗时随粒子数近似线性，GA 斜率明显更高
- 图表可放入 PPT

### 报告写法

> MEVisor 采用 GA，个体评估与遗传算子存在串行依赖。本项目以向量化 PSO 替换后，在 N=1000 粒子时加速约 ___ 倍（填实测数据）。

## E2：真多 GPU 并行加速

### 目标

在 **真实多卡环境** 验证 PSO 粒子分片后的墙钟加速。

### 前置条件

- Linux 云主机，2 或 4 张 NVIDIA GPU（推荐 AutoDL RTX 4090 多卡实例）
- PyTorch + CUDA + `nccl` 可用
- 已实现 `scripts/run_ddp.py` 与 `optimizer` 内 gbest 同步

### 输入

- 固定 Mock 快照 1 条（排除监听变量）
- 总粒子数 1000，迭代 100
- GPU 数量：1 / 2 / 4

### 命令（待代码实现后使用）

```bash
# 2 卡
torchrun --nproc_per_node=2 scripts/run_ddp.py --mock --record-id snap_003

# 4 卡
torchrun --nproc_per_node=4 scripts/run_ddp.py --mock --record-id snap_003

# 批量 benchmark
python scripts/benchmark_multi_gpu.py --gpus 1,2,4 --particles 1000
```

### 产出


| 文件                                      | 内容             |
| --------------------------------------- | -------------- |
| `data/figures/multi_gpu_speedup.png`    | 1/2/4 卡墙钟耗时柱状图 |
| `data/figures/multi_gpu_efficiency.png` | 加速比与并行效率折线     |
| `data/results/multi_gpu_benchmark.json` | 原始耗时数据（可复现）    |


### 验收标准

- 使用 `torch.distributed`，非 CPU gloo 假装多卡
- 报告注明云主机型号、GPU 数量、PyTorch/CUDA 版本
- 2 卡耗时 < 1 卡耗时（允许非完美 2×，但须有明显加速）
- 附 `nvidia-smi` 截图证明多卡同时工作

### 报告写法

> 将 1000 粒子均分至 K 张 GPU，每轮迭代通过 all_reduce 同步全局最优。实测 1→2→4 卡墙钟时间为 ___ / ___ / ___ ms，加速比分别为 1.0× / ___× / ___×。

### 租卡参考

见 [环境配置指南.md](./环境配置指南.md) 第 8 节。预算约 **15～30 元**（1～2 小时）。

## E3：L1-L2 跨层套利检测

### 目标

当 L1 发生大额 Swap 或价差满足条件时，触发 PSO 搜索 L2 对冲路径，输出可盈利策略。

### 子实验

#### E3a：Mock 回放（必做）

```bash
python main.py --mock
python main.py --mock --record-id snap_005
```

**产出：** 每条快照的价差、策略摘要、盈利/无机会统计。

#### E3b：实时监听（必做）

```bash
# 国内网络推荐 HTTP 模式
python main.py --live --http --duration 3600
```

**产出：** 1～2 小时监听日志，至少 1 个触发案例（或说明市场无价差并附截图）。

#### E3c：L1/L2 自适应 PSO（建议）

- 实现 `select_pso_profile(snapshot)` → 动态 `num_particles` / `max_iter`
- 对比固定参数 vs 自适应参数在同一批 Mock 上的耗时与利润

### 验收标准

- Mock 10 条快照全部可跑通
- 至少 3 条 Mock 检出正利润（`mock_mempool.json` 已设计）
- Live 模式可稳定连接 Arbitrum；Ethereum 可用 HTTP 回退
- 输出含：`amount_in`、路由、Gas、桥费、净利润、搜索耗时

## E4：检测与可视化系统

### 目标

提供可演示的 **可视化面板**，展示跨层价差、触发记录、搜索结果与算法性能，不仅是 CLI 日志。

### 技术选型

**Streamlit**（`viz/dashboard.py`），读取：

- 实时：监听进程写入的 `data/results/live_log.jsonl`
- 离线：Mock 回放结果 / benchmark JSON

### 面板模块


| 模块   | 内容                            |
| ---- | ----------------------------- |
| 价差曲线 | L1 vs L2 ETH/USDC 价格、价差%      |
| 触发列表 | snapshot_id、块高、l2_lag_ms、触发原因 |

### 命令

```bash
streamlit run viz/dashboard.py
```

### 验收标准

- 可本地一键启动，答辩时可投屏演示
- 至少展示 Mock 回放数据
- 含 E1 或 E2 性能图至少一张
