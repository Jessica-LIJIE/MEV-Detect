# 最小闭环实施方案

> 依据 [项目说明.md §7 老师反馈版](./项目说明.md#7-老师反馈版课题重定位) 编写。  
> 目标：在课程周期内交付**可演示、可写进报告、与老师框架对齐**的最小可行版本（MVP），而非复现论文级完整系统。

## 1. 范围界定

### 1.1 最小闭环要证明什么

| 必须证明 | 刻意不做（报告写明） |
|----------|----------------------|
| 跨层**多池**路由比单池闭式问题更复杂、更值得搜索 | 完整负对数环检测 + 全图 Bellman-Ford |
| 两阶段框架可跑通：候选集 → PSO 资金分配 | 所有代币的通用环形路径枚举 |
| 真实 RPC 快照 + Mock 消融**分开** | Mock 利润当作「市场发现」 |
| 至少 1～2 条路由经 **fork / QuoterV2** 验真 | 全量历史回放 3941 条案例 |
| E1 含闭式上界锚点；E2 含单卡规模曲线 | 强行多卡出加速比 |
| E4 威胁与限定文字完整 | E4 用 Streamlit 代替 |

### 1.2 阶段1 的刻意简化（MVP）

老师框架中的「负对数汇率 + 可采纳松弛环检测」在 MVP 中简化为：

**跨层笛卡尔候选集 + 深度过滤：**

```text
C = { (pool_l1[i], bridge_b, pool_l2[j]) | i∈[0,P1), j∈[0,P2), b∈[0,B) }
过滤：effective_depth(pool) ≥ min_depth
排序：静态 score = spread_ij × min(liq_l1, liq_l2)
保留 Top-K（建议 K=32～64）
```

报告表述为：「阶段1 的可采纳松弛之 MVP 实现；完整环检测列为 future work。」

### 1.3 阶段2 的刻意简化（MVP）

- 搜索变量：`amount_in`（连续）+ `candidate_idx`（离散，0..K-1）+ 可选 `bridge_idx`
- Fitness：**批量**评估 `[N_particles, K]` 利润矩阵（GPU 友好），再按 `candidate_idx` 取对应列
- 库存约束（MVP）：`amount_in ≤ min(inventory_l1_eth, inventory_l2_eth)`
- 时延风险（MVP）：`fitness -= λ × bridge_latency_hours × amount_in × price × volatility_proxy`

## 3. 分阶段实施（建议 2～3 周）

### Phase 0：配置与类型（1～2 天）✅

**任务**

1. 编写 `data/pool_registry.json`：L1/L2 各 **6 个**高流动性 Uniswap V3 池（WETH/USDC 0.05%/0.3%、WETH/USDT、WETH/DAI 等）。
2. 扩展 `config/settings.py`：
   - `POOL_REGISTRY_PATH`
   - `BRIDGE_CONFIG`（慢桥 / Across / Hop 的 fee、latency_hours）
   - `INVENTORY_DEFAULT`（l1_eth, l2_eth）
   - `CYCLE_FINDER_CONFIG`（`top_k`, `min_effective_depth`）
3. 在 `src/types.py` 增加：

```python
@dataclass
class PoolSnapshot:
    chain: str
    pool_address: str
    token0: str
    token1: str
    fee_tier: int
    sqrt_price_x96: int
    liquidity: int
    tick: int
    eth_usdc_price: float  # 统一报价便于跨池比较
    ticks_loaded: bool = False  # MVP 主池可为 True

@dataclass
class MultiPoolSnapshot:
    snapshot_id: str
    l1_block: int
    l2_block: int
    l1_batch: int | None          # Arbitrum → L1 batch 映射结果
    l1_pools: list[PoolSnapshot]
    l2_pools: list[PoolSnapshot]
    gas: GasState
    bridge_fee_usd: float
    inventory: InventoryState
    trigger: str
    timestamp: str

@dataclass
class CandidateRoute:
    route_id: int
    l1_pool_idx: int
    l2_pool_idx: int
    bridge_idx: int
    score: float                  # 阶段1 静态分
```

**验收**

```bash
python -c "from src.pool_registry import load_registry; print(len(load_registry('ethereum')))"
# 期望：L1、L2 各 6
```

### Phase 1：真实快照采集（2～3 天）✅

**任务**

1. 实现 `src/pool_registry.py`：Multicall3 `aggregate3` 批量 `slot0()` + `liquidity()`。
2. 实现 `src/snapshot_builder.py`：
   - 所有 call 钉死 `block_identifier`
   - 记录 `l2_block` 与 `l1_batch`（MVP：调用 ArbSys 或文档约定的 batch 查询；失败则记录 `null` 并在 E3 说明）
3. 实现 `scripts/fetch_multi_pool_snapshot.py`：

```bash
python scripts/fetch_multi_pool_snapshot.py \
  --l1-block 20100100 \
  --l2-block 220001500 \
  --out data/snapshots/snap_real_001.json
```

**MVP tick 策略**

| 优先级 | 池 | tick 数据 |
|--------|-----|-----------|
| P0 | 每链 WETH/USDC 0.05% 主池 | 拉 tickBitmap + 邻近 ticks（或 QuoterV2） |
| P1 | 其余池 | 仅 slot0 + liquidity；大 swap 用简化 slippage |

**验收**

- 输出 JSON 含 ≥6+6 池状态
- 同一 `l1_block` 重复拉取结果一致

### Phase 3：阶段2 批量 fitness + 库存/时延（2～3 天）✅

**任务**

1. 重写 `src/models.py` 中 `CostModel.evaluate_fitness`：

```text
positions: [N, D]  其中 D=2: amount_in, candidate_idx（连续编码后 floor）
对每个粒子 p、候选 k：
  profit[p,k] = simulate_swap_path(amount_p, candidate_k, snapshot)
fitness[p] = max_k profit[p,k] - gas - bridge - λ·latency·risk
             （或 fitness[p] = profit[p, idx_p] 若粒子绑定 candidate_idx）
```

**MVP 推荐：** 粒子维 `[amount, cand_idx_float]`，fitness 取 `profit[p, floor(cand_idx)]`，同时保留 `max_k` 作诊断。

2. 实现 `src/swap_simulator.py`：
   - `estimate_amount_out(amount_in, pool, ...)` 常数乘积 + 费率 + 简化 V3 滑点
   - `quote_via_quoter_v2(...)` 可选，供 fork 前校验
3. 实现 `src/closed_form.py`：对**单条**候选环做 amount 三元搜索/黄金分割，作为 E1 **质量上界**

**验收**

```bash
python -m pytest tests/test_batch_fitness.py -v
# 100k 粒子 × K=32：单卡 CUDA 墙钟 > 500ms（说明规模够做 E2）
```

### Phase 5：fork 验真（2～3 天）✅

**任务**

1. `scripts/fork_verify_route.py`：
   - 启动或连接 `anvil --fork $ETH_RPC --fork-block-number $L1_BLOCK`

**验收**

- 至少 1 条路由 `rel_error < 10%`（报告给出绝对值与 block 号）

### Phase 6：实验 E1～E4 重跑（3～4 天）✅

#### E1 算法对照

**改** `tests/test_pso_vs_ga.py`：

| 选手 | 说明 |
|------|------|
| 向量化 PSO | 固定迭代 |
| 向量化 GA | 同迭代、同 K |
| 闭式/三元上界 | 对每条候选环求最优 amount，取 max 作 quality anchor |

**指标**

- 墙钟时间（预热 3 次后 mean±std）
- `quality_ratio = pso_profit / closed_form_profit`

**命令**

```bash
python tests/test_pso_vs_ga.py --snapshot data/mock_multipool/records.json --record-id mock_003
```

## 6. 风险与降级策略

| 风险 | 降级 |
|------|------|
| tick 全量太贵 | 主池 QuoterV2 + 其余池简化 slippage |
| batch 映射 API 不稳 | 记录 `l2_block` + 时间戳，E3 说明对齐近似 |
| anvil fork 环境搭不起来 | 仅用 QuoterV2 staticcall 作验真，报告注明未做 state-changing fork |
| 多卡仍负加速 | 按老师要求写进 E2：「问题规模在 X 以下多卡不适用」 |
| 时间不够 | 优先 Phase 3+4+5 各 1 条；E2 只做单卡曲线 |

## 7. 报告中的标准表述（可直接引用）

> 本文将跨层非原子多池套利形式化为带分层库存与时延风险的优化问题，采用「候选环筛选 + 向量化 PSO 资金分配」两阶段框架。实验数据来自 RPC 自建多池快照（钉块、含主池 tick/Quoter 校验），Mock 仅用于算法消融且与真实结果分报告。利润以主网分叉回放与 QuoterV2 抽样校验为准；本文定位为机会检测，不主张竞争环境下必然捕获 MEV。
